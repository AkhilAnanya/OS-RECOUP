Synchronization

Synchronization refers to the coordination of multiple processes or threads to ensure correct and predictable execution when accessing shared resources.

Critical Section Problem

A critical section is a code segment that accesses shared variables and must not be concurrently executed by more than one process/thread. The critical section problem involves designing a protocol that processes can use to coordinate their access to critical sections.

Solution requirements for critical section problem:

  1. Mutual Exclusion: Only one process can be in its critical section at a time
  2. Progress: If no process is in its critical section, then only processes not in their remainder section can decide which enters next
  3. Bounded Waiting: There must be a bound on the number of times other processes can enter their critical sections after a process has requested entry

Peterson's Solution

A software-based solution for two processes that satisfies all three requirements:

// For process Pi
do {
    flag[i] = true;
    turn = j;
    while (flag[j] && turn == j);
    
    // Critical Section
    
    flag[i] = false;
    
    // Remainder Section
} while (true);

Synchronization Hardware

Hardware solutions for synchronization:

  • Memory Barriers: Ensure load and store operations complete in order
  • Test-and-Set: Atomic instruction that tests and modifies a memory location
  • Compare-and-Swap: Atomic instruction that compares and swaps memory content

Semaphores

A semaphore is an integer variable that, apart from initialization, is accessed only through two atomic operations: wait() and signal().

// Definition of wait()
wait(S) {
    while (S <= 0); // busy wait
    S--;
}

// Definition of signal()
signal(S) {
    S++;
}

Types of semaphores:

  • Counting Semaphore: Integer value can range over an unrestricted domain
  • Binary Semaphore: Integer value can range only between 0 and 1

Classical Problems of Synchronization

Several classic problems demonstrate various aspects of process synchronization:

1. Bounded-Buffer Problem: Also known as the producer-consumer problem, where producers insert items into a buffer and consumers remove them.

2. Readers-Writers Problem: A data set is shared among several concurrent processes where readers only read and writers can both read and write.

3. Dining Philosophers Problem: Five philosophers sit around a table with five chopsticks, alternating between thinking and eating.

Threads

A thread is a basic unit of CPU utilization; it comprises a thread ID, a program counter, a register set, and a stack. Threads within a process share code section, data section, and OS resources.

Benefits of Multithreading

  • Responsiveness: Allows a program to continue running even if part of it is blocked
  • Resource Sharing: Threads share memory and resources of their process
  • Economy: Creating threads is cheaper than creating processes
  • Scalability: Can take advantage of multiprocessor architectures

User Threads vs Kernel Threads

Feature User Threads Kernel Threads
Implementation Managed by user-level threads library Supported by OS kernel
Creation/Management Fast (no system calls) Slow (requires system calls)
Blocking If one thread blocks, all threads block If one thread blocks, others can run
Examples POSIX Pthreads, Java threads Windows, Linux threads

Multithreading Models

There are three common ways to establish relationship between user and kernel threads:

1. Many-to-One Model: Maps many user-level threads to one kernel thread.

2. One-to-One Model: Maps each user thread to a kernel thread.

3. Many-to-Many Model: Multiplexes many user-level threads to a smaller or equal number of kernel threads.

Thread Libraries

Thread libraries provide programmers with an API for creating and managing threads:

  • POSIX Pthreads: Standardized thread API for UNIX systems
  • Windows Threads: Thread API for Windows OS
  • Java Threads: Thread management through the Java language

Threading Issues

Several issues arise with the use of threads:

  • fork() and exec() System Calls: Behavior varies across systems when threads call these
  • Thread Cancellation: Terminating a thread before it has finished
  • Signal Handling: Which thread should receive signals sent to a process
  • Thread Pools: Creating a number of threads at process startup
  • Thread-Specific Data: Data that needs to be unique to each thread

Deadlock

A deadlock is a situation where a set of processes are blocked because each process is holding a resource and waiting for another resource acquired by some other process.

Necessary Conditions for Deadlock

All four conditions must hold simultaneously for a deadlock to occur:

  1. Mutual Exclusion: At least one resource must be held in a non-sharable mode
  2. Hold and Wait: A process must be holding at least one resource and waiting for additional resources
  3. No Preemption: Resources cannot be preempted; they must be released voluntarily
  4. Circular Wait: A set of waiting processes must exist where each is waiting for a resource held by the next

Resource-Allocation Graph

A directed graph used to visualize resource allocation and requests:

  • Processes: Represented as circles
  • Resources: Represented as rectangles with dots indicating instances
  • Request Edge: Directed edge from process to resource
  • Assignment Edge: Directed edge from resource instance to process

If the graph contains no cycles, then no deadlock exists. If there is a cycle, a deadlock may exist.

Methods for Handling Deadlocks

There are three principal approaches:

  1. Deadlock Prevention: Ensure that at least one of the necessary conditions cannot hold
  2. Deadlock Avoidance: Requires additional information about how resources will be requested
  3. Deadlock Detection and Recovery: Allow deadlocks to occur, detect them, and recover

Deadlock Prevention

Attacking each of the four necessary conditions:

  • Mutual Exclusion: Make resources sharable when possible
  • Hold and Wait: Require processes to request all resources before execution
  • No Preemption: If a process can't get a resource, release all held resources
  • Circular Wait: Impose a total ordering of resource types

Deadlock Avoidance

Requires that the system has some additional a priori information available:

  • Safe State: A state is safe if the system can allocate resources to each process in some order and still avoid a deadlock
  • Resource-Allocation Graph Algorithm: For single instances of each resource type
  • Banker's Algorithm: For multiple instances of each resource type

Banker's Algorithm

Used to determine if a system is in a safe state:

  1. Let Work and Finish be vectors of length m and n respectively
  2. Initialize Work = Available and Finish[i] = false for i = 0, 1, ..., n-1
  3. Find an index i such that both:
    • Finish[i] == false
    • Need_i â‰¤ Work
  4. If no such i exists, go to step 5
  5. Work = Work + Allocation_i
  6. Finish[i] = true
  7. Go to step 3
  8. If Finish[i] == true for all i, then the system is in a safe state

Deadlock Detection

If neither avoidance nor prevention is implemented, deadlocks can occur and must be detected:

  • Single Instance of Each Resource Type: Use a wait-for graph
  • Several Instances of a Resource Type: Use an algorithm similar to the Banker's algorithm

Recovery from Deadlock

Once a deadlock is detected, the system must recover:

  • Process Termination:
    • Abort all deadlocked processes
    • Abort one process at a time until the deadlock cycle is eliminated
  • Resource Preemption:
    • Select a victim - which resources to preempt
    • Rollback - return the process to some safe state
    • Starvation - ensure the same process is not always the victim

Synchronization MCQs

1. What is the main purpose of process synchronization?
  • A) To increase the speed of process execution
  • B) To coordinate access to shared resources among competing processes
  • C) To reduce the memory usage of processes

12. A monitor is:
  • A) A hardware synchronization device
